## アーキテクチャ改善ノート

GPT による正直レビューから抽出したコードベースの技術的改善ポイント。機能追加と並行して対応すると後工程が楽になる。

### 1. 状態管理の導入（zustand）

**現状の課題**  
`App.tsx` がサーバーリスト・選択中サーバー・現在ビューなど多くの状態を `useState` で抱えており、Props のバケツリレーが発生しやすい構造。

**改善案**  
`zustand` を導入してグローバルストアを分割管理する。

```text
store/
├── serverStore.ts     # サーバーリスト・選択状態
├── consoleStore.ts    # ログバッファ・フィルタ設定
├── uiStore.ts         # currentView・サイドバー開閉状態
└── settingsStore.ts   # アプリ設定・テーマ
```

---

### 2. ログストリーム改善（リアルタイム性向上）

**現状の課題**  
`src-tauri/src/commands/server.rs` の stdout ストリームを1行ずつ `emit` しているが、高負荷時に UI 側でのバッファリング・デスロットルがないため描画コストが高くなる懸念がある。

**改善案**

- フロント側で50ms間隔でバッファをフラッシュするデスロットル処理を追加
- ログ行に `level` タグを付与して emit することでフィルタリングをバックエンドで前処理できるようにする

---

### 3. Server State Machine（起動管理の安定化）

**現状の課題**  
サーバーの状態が `isRunning` フラグのみで管理されており、「起動中」「停止中」「再起動待ち」「クラッシュ」などの中間状態を表現しにくい。

**改善案**  
明確な State Machine を Rust 側で定義する：

```
Stopped → Starting → Running → Stopping → Stopped
                               ↓
                            Crashed → Restarting → Starting
```

フロント側のボタン制御も State に基づいてアクティブ/無効を切り替えることで誤操作が防止できる。

---

### 4. Plugin API 統合レイヤー

**現状の課題**  
プラグイン・Mod のダウンロードは `download.rs` の汎用コマンド任せであり、Modrinth / Hangar / CurseForge 各 API のレスポンス構造の差異を吸収するアダプター層がない。

**改善案**  
`src/lib/plugin-commands.ts` に統一インターフェースを定義し、API ソースを抽象化する：

```ts
interface PluginSource {
  search(query: string, gameVersion: string): Promise<PluginResult[]>;
  getVersions(projectId: string): Promise<PluginVersion[]>;
  download(versionId: string, targetDir: string): Promise<void>;
}
```

各 API（Modrinth / Hangar / CurseForge）はこのインターフェースを実装するアダプターとして追加できる構造にする。
